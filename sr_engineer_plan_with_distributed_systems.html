Track 1B — from Design Gurus · what to study and in what order
This is the gap your original plan had. These topics come up in Lead/Architect interviews at banks, telcos, and consulting firms. You don't need to go as deep as a software infrastructure engineer — but you need conversational fluency and the ability to apply these concepts to data platform design.
Priority 1 Must know — directly relevant to data platform interviews
Critical gap
CAP theorem & consistency models
Strong vs eventual consistency, PACELC, when to choose each. Interviewers ask: "Is your pipeline strongly consistent? What happens if a node fails mid-write?" You must answer this fluently.
Critical gap
Caching strategies
Read-through, write-through, write-behind, cache-aside, cache invalidation, TTL. Data architects must know when caching fits into the serving layer and what breaks when it fails.
Critical gap
Kafka internals (deep)
Topics, partitions, consumer groups, offset management, delivery semantics (at-least-once, exactly-once), ZooKeeper role. Design Gurus has a full Kafka case study — work through all of it.
Critical gap
Batch vs stream processing trade-offs
When Lambda (batch + speed layer) beats Kappa (unified streaming), latency vs throughput, micro-batch vs true streaming. Directly maps to Azure Databricks Structured Streaming decisions.
Important
Database replication & sharding
Primary-replica, peer-to-peer, read replicas, sharding strategies (range, hash, directory). How these map to Databricks cluster design and ADLS partitioning decisions.
Important
Fault tolerance patterns
Heartbeat, leader/follower, quorum, checkpointing. How Spark checkpointing to ADLS maps to these general patterns. Interviewers probe: "What happens if your streaming job fails at exactly this point?"
Priority 2 Good to know — for Lead/Architect depth credibility
Load balancing
Algorithms (round-robin, least-connections, weighted), stateful vs stateless. Relevant when designing data API serving layers or multi-cluster Databricks setups.
API gateway patterns
Rate limiting, authentication, request routing. Comes up when designing data product APIs or exposing Delta Sharing endpoints to consumers.
Message queue patterns
RabbitMQ vs Kafka vs Azure Service Bus — know the differences and when each fits. Maps to your Azure Event Hubs experience.
SQL vs NoSQL trade-offs
ACID vs BASE, when to use each. Deepens your existing Snowflake vs MongoDB vs PostgreSQL decision-making story for interview discussions.
Consistent hashing
How data is distributed across nodes without full reshuffling. Relevant to partitioning strategy discussions in large-scale Databricks clusters.
Design Gurus case study: Uber
Practice one full product design (Uber or Twitter). Not because you'll design ride-sharing — but because interviewers use product designs to test distributed systems reasoning.
How to use Design Gurus for this track: Their "System Design Fundamentals" and "System Design Trade-offs" sections cover Priority 1. The Kafka case study is the single most valuable thing to read. Budget 3–4 hours/week for 4 weeks on this track running in parallel with Track 1A. Free tier is enough — don't pay until you've exhausted the free content.
