Functional Requirements — What the system must DO

Interview Framework
Before drawing any box, spend 3–5 minutes asking clarifying questions. Interviewers explicitly score this — it shows architect-level thinking.

Clarifying Questions to Always Ask

  • Data volume: How many records/events per day? Peak vs average?
  • Latency SLA: Real-time (<1s), near-real-time (1–60s), or batch (hours)?
  • Sources: How many upstream systems? Push or pull model?
  • Consumers: Who reads the data? BI, ML, APIs? Their SLAs?
  • Data retention: Hot/warm/cold data strategy? How long?
  • Consistency: Strong consistency required or eventual acceptable?
  • Compliance: GDPR, HIPAA, PCI-DSS requirements?
  • Budget: Cost-sensitive or performance-first trade-off?

CAP Theorem in Data Engineering

In any distributed system, you can only guarantee 2 of 3: Consistency, Availability, Partition Tolerance. In data engineering, partition tolerance is given — so choose C or A.

Consistency Priority (CP)

  • Delta Lake ACID transactions
  • Strong read-after-write guarantees
  • Use for financial, audit pipelines
  • May sacrifice write throughput
  • Snapshot isolation by default

Availability Priority (AP)

  • Eventual consistency acceptable
  • Higher write throughput
  • Use for analytics, reporting layers
  • Stale reads acceptable for dashboards
  • Kafka at-least-once delivery
Interview Answer
"Delta Lake gives us CP using optimistic concurrency control. For our streaming pipeline, we accept eventual consistency at Event Hubs but ensure exactly-once semantics by the time data reaches Delta via idempotent MERGE. The Gold layer reporting tables can tolerate eventual consistency since BI queries don't require the latest 30-second window."

Full Load Pattern ▼

When to use: Small lookup/reference tables, no reliable change tracking, table <10 GB.

# Full load — truncate and replace atomically
df = spark.read.format("jdbc").options(**jdbc_opts).load()
df.write.format("delta") \
  .mode("overwrite") \
  .option("overwriteSchema", "true") \
  .saveAsTable("catalog.bronze.customers")
Trade-off
Full loads are simple but expensive for large tables. Use only when incremental tracking is unreliable.
Incremental Load with Watermark ▼
# Read last watermark from control table
last_wm = spark.sql("SELECT MAX(wm) FROM control.watermarks WHERE tbl='orders'").collect()[0][0]
df_new = spark.read.format("jdbc") \
  .option("query", f"SELECT * FROM orders WHERE updated_at > '{last_wm}'").load()
# Upsert to Silver
DeltaTable.forName(spark, "catalog.silver.orders").alias("t") \
  .merge(df_new.alias("s"), "t.order_id = s.order_id") \
  .whenMatchedUpdateAll().whenNotMatchedInsertAll().execute()
# Advance watermark
spark.sql("UPDATE control.watermarks SET wm = now() WHERE tbl='orders'")
Auto Loader — Best for ADLS File Ingestion ▼
# Auto Loader — exactly-once file tracking, schema inference
df = spark.readStream.format("cloudFiles") \
  .option("cloudFiles.format", "json") \
  .option("cloudFiles.schemaLocation", "/mnt/schema/orders") \
  .load("abfss://raw@storage.dfs.core.windows.net/orders/")
df.writeStream.format("delta") \
  .option("checkpointLocation", "/mnt/checkpoints/orders") \
  .trigger(availableNow=True) \  # batch mode for backfill
  .toTable("catalog.bronze.orders")

1
Idempotent Writes
Always use MERGE or INSERT OVERWRITE PARTITION so re-runs don't create duplicates. Never use append mode for processed data.
2
Control Table Pattern
Track pipeline state: last_run_start, last_successful_wm, status. On failure, restart from last successful watermark.
3
Dead Letter / Quarantine
Route bad records to quarantine Delta table. Process good records, alert on quarantine growth, reprocess after fix.
4
Delta Time Travel for Rollback
If a bad write gets through, use RESTORE TABLE to roll back. Delta retains 30 days of history by default.
# Restore Delta table to previous version after bad write
spark.sql("RESTORE TABLE catalog.silver.orders TO VERSION AS OF 42")
# Or by timestamp
spark.sql("RESTORE TABLE catalog.silver.orders TO TIMESTAMP AS OF '2024-01-15'")
🌊Streaming Processing

Spark Structured Streaming on Azure Databricks — from basics to production patterns.

graph LR A[Event Hubs / Kafka] --> B[Structured Streaming] B --> C[Parse + Validate] C --> D{Error?} D -->|Bad Record| E[Dead Letter Table] D -->|Good| F[Watermark + Window] F --> G[foreachBatch MERGE] G --> H[Silver Delta] H --> I[DLT Gold] I --> J[Snowflake / Power BI] 
# Production streaming job — Event Hubs → Delta Silver
from pyspark.sql import functions as F
df_raw = spark.readStream.format("eventhubs").options(**eh_conf).load()
df_parsed = df_raw.select(F.from_json(F.col("body").cast("string"), schema).alias("d"), "enqueuedTime").select("d.*", "enqueuedTime")
df_wm = df_parsed.withWatermark("event_time", "10 minutes")
df_wm.writeStream.format("delta") \
  .option("checkpointLocation", "/mnt/checkpoints/events") \
  .trigger(processingTime="30 seconds") \
  .toTable("catalog.silver.events")
Core Concept
Watermark = max(event_time_seen) - watermark_duration. Any event arriving after this threshold is considered late and dropped (or handled separately). Enables Spark to safely free state memory.
Source TypeWatermark DurationRationale
IoT sensors (reliable network)2–5 minutesLow latency, tight SLA
Mobile app events15–30 minutesOffline batching, intermittent connectivity
Log aggregation agents1–2 hoursAgents may buffer and retry
Financial transactions24 hoursEnd-of-day reconciliation window

Short Watermark

  • Less state memory usage
  • State expires sooner
  • Risk of dropping late events
  • Good for real-time dashboards

Long Watermark

  • Higher memory (state store)
  • Tolerates more late-arriving data
  • Higher latency to close windows
  • Good for accuracy-critical pipelines

Exactly-Once = Checkpointing + Idempotent Sink

1
Checkpointing (at-least-once source)
Spark writes processed offsets to ADLS checkpoint directory. On restart, it resumes from last committed offset — no re-processing from beginning.
2
Idempotent Delta Sink
Delta Lake's write-ahead log rolls back partial writes. MERGE ensures same result whether written once or many times. Together = exactly-once.
3
foreachBatch for MERGE
Use foreachBatch to run custom MERGE logic on each micro-batch — this enables upsert (SCD Type 1/2) on streaming data.
# Exactly-once streaming MERGE using foreachBatch
def upsert_to_delta(microBatchDF, batchId):
    microBatchDF.createOrReplaceTempView("updates")
    microBatchDF._jdf.sparkSession().sql("""
      MERGE INTO catalog.silver.customers t
      USING updates s ON t.customer_id = s.customer_id
      WHEN MATCHED THEN UPDATE SET *
      WHEN NOT MATCHED THEN INSERT *
    """)

df_parsed.writeStream \
  .foreachBatch(upsert_to_delta) \
  .option("checkpointLocation", "/mnt/checkpoints/customers") \
  .trigger(processingTime="1 minute").start()
AspectMicro-batch (default)Continuous Processing
Latency100ms – minutes (configurable)<1ms
ThroughputHigh (batching efficiency)Lower

Interview Answer
"In practice I always use micro-batch with short trigger intervals (30s–5min). True continuous processing sacrifices exactly-once and is experimental in Spark. For sub-second latency requirements I'd evaluate Azure Stream Analytics or Event Hubs → Function App → Cosmos DB as an alternative."

The attribute never changes once loaded. Represents immutable historical facts.

Type 4 — History Table (Separate)

Current values in main table. All history in a separate audit/history table. Best query performance for both current and historical lookups.

📋
Main Table (current only)
One row per entity, always current. Fast lookups. Used by most operational queries. No joins needed for current state.
📚
History Table (append-only)
All changes logged here. Used for auditing and point-in-time analysis. Delta CDF naturally produces this pattern.

✅Data Quality

Building production-grade data quality frameworks on Databricks and Delta Lake.

📊
Completeness
All expected records present. NULL rate, row count vs expected, required fields filled. Alert if daily count drops >5%.
🔁
Uniqueness
No duplicate primary keys. Use ROW_NUMBER() or MERGE on business key to deduplicate. Track duplicate rate over time.
🎯
Accuracy
Values within expected range. Reference data validation. Cross-system reconciliation. Amount > 0, age 0–120.
⏰
Freshness
Data arrives within SLA. Alert if max(updated_at) > now() - 2h. Track pipeline lag continuously.
🔗
Consistency
Referential integrity — FK exists in parent table. Cross-table row count reconciliation. Sum of parts = total.
📋
Validity
Values conform to expected format/domain. Email regex, date format, enum values, numeric range, ZIP code pattern.

def run_dq_checks(df, table_name):
    total = df.count()
    checks = {
      "null_rate_cust_id": df.filter(F.col("customer_id").isNull()).count() / total,
      "duplicate_rate": 1 - df.dropDuplicates(["order_id"]).count() / total,
      "pct_negative_amount": df.filter(F.col("amount") < 0).count() / total

    # Write metrics to monitoring Delta table

    spark.createDataFrame(dq_rows, ["table", "check", "value", "status"]) \
      .write.format("delta").mode("append").saveAsTable("catalog.monitoring.dq_metrics")

⏮️Backfill Strategies

Safely reload historical data without breaking production pipelines.

1
Partition-based Backfill
Overwrite each partition independently. Safe — doesn't affect other partitions. Best for date-partitioned tables.
2
Shadow Table Pattern
Write backfill to a shadow table. Validate quality. Swap with RENAME. Zero downtime, safe rollback via Delta time travel.
3
Range-based MERGE Replay
Replay a specific date range using idempotent MERGE. Safe to re-run. Best for SCD Type 2 dimensions.
4
Auto Loader Backfill
Use trigger(availableNow=True) on new checkpoint location to process historical files in batch mode using same streaming code.
# Partition backfill — replaceWhere = atomic, idempotent
spark.conf.set("spark.sql.sources.partitionOverwriteMode", "dynamic")
backfill_df = spark.read.format("delta") \
  .option("versionAsOf", 0) \
  .table("catalog.bronze.orders") \
  .filter("order_date BETWEEN '2023-01-01' AND '2023-12-31'")
backfill_df.write.format("delta") \
  .mode("overwrite") \
  .option("replaceWhere", "order_date BETWEEN '2023-01-01' AND '2023-12-31'") \
  .saveAsTable("catalog.silver.orders")
Definition
An idempotent operation produces the same result whether run once or many times. Essential for safe backfills and retries.
OperationIdempotent?Why
MERGE (upsert)✅ YesSame result whether run 1x or 10x
INSERT OVERWRITE PARTITION✅ YesReplaces partition atomically each time
replaceWhere write✅ YesAtomic replacement of matching rows
INSERT INTO (append)❌ NoCreates duplicates on re-run
Streaming append (no checkpoint)❌ NoRe-processes from beginning = duplicates
Anti-Pattern
Never use append mode for backfills. A failed job that partially ran leaves duplicates. Always use MERGE or replaceWhere.

Backfill Performance Considerations

1
Off-Peak Scheduling
Run backfills during off-peak hours to avoid competing with streaming jobs for cluster resources and ADLS IOPS.
2
Dedicated Backfill Cluster
Use a separate, larger spot-instance cluster for backfill. Don't share with production streaming — isolate the blast radius.
3
Chunk by Partition
Process one month at a time rather than all years at once. Enables progress tracking and partial failure recovery.
4
OPTIMIZE After Backfill
Backfills often create many small files. Run OPTIMIZE + ZORDER after completion to restore query performance.
⚙️Cluster Design

Right-sizing and optimizing Databricks clusters for performance and cost.

All-Purpose Clusters

  • Interactive development
  • Shared by multiple users
  • Higher DBU rate (always-on)
  • Auto-terminate after idle
  • Cannot be shared across Workflow jobs

Job Clusters

  • Created per job run, terminated after
  • Cheapest — no idle cost
  • Best for production pipelines
  • Faster startup with instance pools
  • Use cluster policies for governance

Target 128–200 MB per partition. num_partitions = num_cores × 2–4. Enable AQE to auto-tune at runtime. Always measure Spark UI before scaling up.

1
Spot Instances for Batch (60–80% savings)
Use Azure Spot VMs for batch jobs. Configure fallback to on-demand for driver. Streaming jobs need on-demand to avoid interruptions.
2
Instance Pools
Pre-warm a pool of VMs. Job clusters attach instantly (no 3–5 min spin-up). Reduces total wall time and improves SLA compliance.
3
Photon Engine
Enable for SQL-heavy workloads — 2–5x faster with same or fewer DBUs. Best ROI for MERGE, Z-ORDER, and aggregation operations.
4
Cluster Policies + Auto-Terminate
Enforce size limits, required tags, 30-min auto-terminate on interactive clusters. Prevent runaway costs from oversized dev clusters.
5
Serverless SQL Warehouses for BI
Auto-scales to zero. No idle cost. Perfect for Power BI / Tableau DirectQuery. Pay only per query second.

def validate_file(file_path):
    try:
        df = spark.read.format("parquet").load(file_path)
        actual_count = df.count()
        expected = get_expected_from_manifest(file_path)
        if actual_count < expected * 0.95:  # 5% tolerance
            move_to_quarantine(file_path)

2. A streaming job's checkpoint directory is accidentally deleted. The job restarts. What happens? Hard
ASpark resumes from where it left off using Event Hubs retention
BJob fails immediately with checkpoint not found error
CSpark starts from the latest offset, potentially missing days of unprocessed data
DDelta Lake automatically reconstructs the checkpoint from the transaction log
Without checkpoint, Spark cannot know which offsets were processed. By default it starts from LATEST, skipping all unprocessed data. Recovery options: restore checkpoint from backup (always back them up!), or replay from earliest offset with new checkpoint path and use MERGE to handle duplicates idempotently.
3. What is the key difference between SCD Type 2 and SCD Type 4? Medium
AType 2 stores current + previous value; Type 4 stores only current
BType 2 stores history in the same table with is_current flag; Type 4 separates current and history into two tables
CType 2 overwrites records; Type 4 inserts new rows
DType 4 is the hybrid of Type 1 + 2 + 3
Type 2: One table, multiple rows per entity (one per change), is_current/start_date/end_date columns. Type 4: Two tables — current table (fast lookups, always latest) and a separate history/audit table (append-only). Delta CDF naturally produces Type 4. Type 6 is the hybrid of 1+2+3.
4. Your PySpark job shows 1 executor running for 2 hours while others finish in 10 minutes. What is the most likely cause? Hard
AData skew — one partition has significantly more records than others
BThe cluster auto-scaled down mid-job
CThe shuffle.partitions config is set too low
DPython UDFs are causing garbage collection pressure
One executor running 12x longer = classic data skew. One partition has a hot key with far more records. Fix: Enable AQE skew join optimization (automatic, no code change). If still skewed, use salting: add random salt 0–9 to join key on both sides (explode right side). This distributes one hot partition into 10 balanced ones.
5. withWatermark("event_time", "15 minutes") — what exactly does this mean? Medium
AThe job processes a batch every 15 minutes
BEvents older than 15 minutes are immediately dropped from the stream
CSpark waits up to 15 min beyond the max event_time seen before closing a window and discarding late arrivals
DThe checkpoint is flushed to ADLS every 15 minutes
Watermark = max(event_time_seen) - watermark_duration. Events with event_time < this threshold are late and dropped (or handled separately). This enables Spark to safely free state memory for closed windows. Trigger interval is separate — set via .trigger(processingTime="30 seconds"). These are independent settings.
6. When should you use Liquid Clustering instead of Z-Ordering in Delta Lake? Medium
AWhen your table is smaller than 100 GB
BWhen you need to cluster on more than one column
CFor large tables with frequent data updates where full Z-Order OPTIMIZE is too slow to run regularly
DLiquid Clustering and Z-Ordering are identical — either works the same
Z-Ordering requires a full OPTIMIZE run that gets more expensive as the table grows. At PB scale it can take hours. Liquid Clustering (Databricks 13.3+) incrementally re-clusters only files that need it as a background operation — much more practical for large, frequently-updated tables. For smaller stable tables, Z-Order still works great.

How do you ensure exactly-once in Structured Streaming?
Click to flip →
Formula
Exactly-once = Checkpointing + Idempotent Sink. Checkpoint = processed offsets saved to ADLS (at-least-once source). Idempotent sink = Delta ACID + MERGE ensures same result whether written once or many times.
